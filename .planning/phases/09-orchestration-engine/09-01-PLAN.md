# Plan 09-01: Orchestration Engine & Execution API

Este plan implementa el motor de ejecución de flujos en Deno y expone la API para dispararlos.

## Goals
1. Convertir el `runner.ts` en un orquestador DAG completo.
2. Implementar el paso de datos (`payload`) entre nodos.
3. Crear el endpoint de ejecución manual en el backend.

## Tasks

### 1. Runner: DAG Orchestration
- Actualizar `backend/app/runtime/runner.ts`.
- Implementar algoritmo de ordenamiento topológico (Kahn o DFS) para determinar el orden de ejecución basado en `connections`.
- Implementar el bucle de ejecución:
    - Inicializar `ctx.payload` con los datos de entrada.
    - Para cada nodo en el orden calculado:
        - Si es `js_script`, ejecutar el código con el `ctx`.
        - Si es un nodo de sistema (ej: `postgresql`), emitir un log indicando que la integración real con estos nodos será en el futuro, pero permitir que el flujo continúe.
        - Actualizar `ctx.payload` con el resultado del nodo.
- Retornar el `payload` final como JSON en `stdout`.

### 2. Deno Service: Run Integration
- Actualizar `backend/app/services/deno_service.py`.
- Añadir método `run_flow(flow_data: dict)`.
- El método debe preparar el JSON completo (nodos, conexiones, payload inicial) y llamar a `execute_deno`.

### 3. API: Execution Endpoint
- Actualizar `backend/app/api/endpoints/integration_flows.py`.
- Añadir `POST /{flow_id}/run`.
- El endpoint debe:
    - Cargar el flujo de la BD.
    - Llamar a `deno_service.run_flow`.
    - Retornar el resultado de la ejecución (logs, errores, payload final).
    - Actualizar `last_run` y `last_run_success` en la BD.

## Verification
1. Crear un flujo con 2 nodos de script conectados:
    - Nodo 1: `return { val: ctx.payload.x + 1 };`
    - Nodo 2: `return { result: ctx.payload.val * 2 };`
2. Ejecutar el flujo vía API con `{"x": 10}`.
3. Verificar que el resultado es `{"result": 22}`.
4. Verificar que los logs muestran la ejecución de ambos nodos en orden.
