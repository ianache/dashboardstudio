<objective>
Pass the active dashboard filter state (CubeJS filter objects) as context with every AI chat request so the agent queries the same data the user sees on screen.

Purpose: When a user has filtered the dashboard to "Region: Lima, April 2026" and asks "what drove the drop in sales?", the agent must analyze April 2026 Lima data — not the full dataset.

Output: Four files modified. Frontend reads resolved filters and forwards them; backend injects them into the user prompt and merges them into query_data tool calls.

**ai-analyst/app/main.py — chat() endpoint:**
After building the `prompt` string (after the `[CONTEXT]` injection block), append a compact filter string if filters are present. Also store active filters in a module-level variable so `cube.py` can read them during the same request:

```python
import app.tools.cube as cube_tool  # at top of file

# Inside chat() endpoint, after prompt construction:
if request.filters:
    filter_parts = []
    for f in request.filters:
        member = f.get("member", "")
        op = f.get("operator", "")
        vals = f.get("values", [])
        filter_parts.append(f"{member} {op} {vals}")
    filter_str = "[ACTIVE FILTERS] " + "; ".join(filter_parts)
    prompt = filter_str + "\n\n" + prompt
    cube_tool._active_filters = request.filters
else:
    cube_tool._active_filters = None
```

**ai-analyst/app/tools/cube.py:**
Add a module-level variable and merge logic:
```python
# Module-level variable set per request by main.py
_active_filters: list | None = None

async def query_data(query: dict):
    """... existing docstring unchanged ..."""
    # Merge active dashboard filters into the query
    if _active_filters:
        existing = query.get("filters", [])
        query = {**query, "filters": existing + _active_filters}

    # ... rest of function unchanged (JWT, httpx call, etc.)
```

Note: Using a module-level `_active_filters` variable is safe because FastAPI handles one request at a time within the async event loop for a single coroutine call chain. This avoids threading issues given the single-process nature of the service.
